# Virtual Memory

Virtual memory is a memory management technique that allows programs to use a large logical address space even when the available physical memory (RAM) is limited.

Each process is given its own **virtual address space**. The addresses used by the program are virtual addresses, and these must be translated into physical addresses before accessing RAM.

This translation is handled by hardware with support from the operating system.

Virtual memory provides several important capabilities:

- Programs can use more memory than physically available
- Processes are isolated from each other
- Memory protection becomes easier
- Memory can be used more efficiently

## Virtual address space

Every process runs in its own virtual address space. From the program's perspective, memory appears as a continuous range of addresses starting from zero.

A typical virtual address space contains:

- Code segment that stores program instructions
- Data segment that stores global variables
- Heap used for dynamic memory allocation
- Stack used for function calls and local variables

Even though the virtual memory appears continuous, the actual physical memory locations may be scattered across different frames in RAM.

## Virtual vs physical addresses

Programs generate **virtual addresses** when accessing memory. These addresses cannot directly reference RAM.

Instead, they must be translated into **physical addresses**.

Definitions:

- Virtual address: address generated by the CPU
- Physical address: actual location in RAM
- Address translation: conversion of a virtual address into a physical address

This translation is performed by the **Memory Management Unit (MMU)**.

## Paging

Virtual memory is usually implemented using **paging**.

Memory is divided into fixed size blocks:

- **Pages** in virtual memory
- **Frames** in physical memory

A page can be stored in any available frame in RAM.

Advantages of paging:

- Eliminates external fragmentation
- Simplifies allocation of memory
- Allows flexible mapping between virtual and physical memory

## Page tables

A **page table** stores the mapping between virtual pages and physical frames.

Each process typically has its own page table.

A page table entry usually contains:

- Frame number
- Valid or invalid bit
- Protection bits
- Dirty bit indicating whether the page was modified
- Reference bit indicating recent usage

Example mapping:

Virtual page → Physical frame

0 → 5  
1 → 12  
2 → 3  

The operating system consults the page table during address translation.

## Address translation

When a program accesses memory, the following steps occur:

1. The CPU generates a virtual address
2. The MMU extracts the page number and offset
3. The page table is consulted to find the corresponding frame
4. The physical address is constructed
5. Memory access occurs in RAM

If the required page is not in RAM, a page fault occurs.

## Translation Lookaside Buffer (TLB)

Page table lookups for every memory access would be slow. To improve performance, systems use a **Translation Lookaside Buffer (TLB)**.

The TLB is a small and fast cache that stores recently used address translations.

Operation:

- If the translation is found in the TLB, it is a **TLB hit**
- If the translation is not found, it is a **TLB miss**, and the page table must be checked

Benefits:

- Reduces the time required for address translation
- Improves memory access performance

## Demand paging

Demand paging loads pages into memory only when they are actually needed.

Instead of loading the entire program into RAM at once, pages are loaded when the program accesses them.

Steps:

1. Program attempts to access a page
2. The page is not in memory
3. A page fault occurs
4. The operating system loads the page from disk
5. Execution resumes

Demand paging allows programs larger than physical memory to run.

## Page faults

A **page fault** occurs when a process accesses a page that is not currently present in physical memory.

Handling a page fault involves the following steps:

1. Hardware detects the missing page
2. Control transfers to the operating system
3. The OS locates the required page on disk
4. The page is loaded into a free frame
5. The page table is updated
6. The interrupted instruction resumes

Because disk access is slow compared to RAM, page faults are expensive operations.

## Page replacement

If physical memory is full and a new page must be loaded, the system must remove an existing page.

This decision is handled by **page replacement algorithms**.

Common algorithms include:

**FIFO (First In First Out)**

- Removes the oldest page in memory

**LRU (Least Recently Used)**

- Removes the page that has not been used for the longest time

**Optimal replacement**

- Removes the page that will not be used for the longest time in the future
- Used mainly for theoretical analysis

## Thrashing

Thrashing occurs when the system spends most of its time swapping pages between disk and memory instead of executing programs.

Symptoms include:

- Very high page fault rate
- Low CPU utilization
- Heavy disk activity

Thrashing typically happens when too many processes compete for limited memory.

Reducing the number of active processes or increasing physical memory can help prevent thrashing.
